RTP Timestamps

RTP timestamps are part of the RTP protocol. Depending on what protocol used, the clock used to genereate the RTP timestamp has other frequencies.
Example Clocks:
  • SMPTE ST-2110: 90kHz
  • SMPTE ST-2022-6 : 27MHz
In the case of 2110 the timestamp-header has a size of 32Bit, so the highest possible value is 4294967296. So based on a 90kHz clock the timestamp will overflow after ca. 13,25h.

If stairs are visible, the sender has problems with its timestamping and thus creates bursty signals. ALso for 2110: all RTP timestamps of a frame have to be equal! So the frame chart should show a peak of 1800 ticks at the first packet, and after that zero for the rest of the frame.

Receiver Timestamps

There are two main ways of adding timestamps to a packet when it is received by a Receiver
  • Hardware timestamping
  • Software timestamping
In hardware timestamping the network card can use a clock to directly timestamp a packet at its arrival and thus provide the best and most accurate timestamp possible. But the NIC has to be capable of this. With software timestamping the timestamps are generated by the operating system. They are usually not as accurate as the packet has to travel through the network stack first before timestamping, which could introduce jitter.
The timestamps in a pcap file are written in seconds till the particular epoch with a temporal resolution of either nanoseconds or microseconds. For analysing 2110 streams, a nano seconds resolution is mandatory. The header size is 64 bit, 32 Bit for the seconds and 32Bit for the nano seconds. This makes it hard to analyze this timestamps with usual Software like Excel, as Excel only supports numbers up to 32Bit. RTP Timestamp Offset

The RTP Timestamp Offsets are the difference between an RTP timestamp to its previous RTP timestamp. This should be useful to detect if the timestamping of the sender is correct. Below is an example of a good capture

Receiver Timestamp Offset

This is just another representation of the receiver timestamps, where the difference between the current receiver timestamp to the previous timestamp is displayed. A good capture should show a straight horizontal line, so a constant offset to the previous packet.

Packet loss results in a bigger delta, which is then represented in a relativly big peak. But packet loss is not the only reason for a peak. If gaps are visible, this means either that the receiver is struggeling with recording the packets and is thus introducing jitter, or that the jitter comes from the network itself. RTP-RX Timestamp Delta

This values should show how long each packet needed to get from the sender to the receiver. For this, the receiver timestamp first has to be converted into an absolute timestamp. Here is a very short explaination of the used algorithm:
  • Convert Receiver Timestamp into ticks
  • Modulo the receiver timestamp in Ticks through the size of the RTP timestamp header ( receiverTimestampTicks % 2^32 )
  • Calculate the difference between this timestamp and the RTP timestamp
The resulting values should be somewhere under 1ms. If the offset is around 36s, the sending device is probably locked to UTC and not TAI (whereas the receiver is locked to TAI). If the offset is somewhere around -36 seconds, the receiver is probably locked to UTC and the sender to TAI.
